SECTION 4 — PHYSICS ENGINE SPECIFICATION

── 4.1 GRAVITY VECTOR CALCULATION ──────────────────────────────────────────

For each clinostat mode, compute the instantaneous gravity vector
experienced by the sample at every timestep dt:

1D MODE:
  g⃗(t) = g₀ · [cos(ω₁t), sin(ω₁t), 0]ᵀ
  Time-averaged: g⃗_avg = (1/T)·∫₀ᵀ g⃗(t)dt → 0 as T → ∞

2D MODE (RPM):
  g⃗(t) = R_y(ω₂t) · R_x(ω₁t) · [0, 0, g₀]ᵀ
  Where ω₁ ≠ ω₂ and ratio ω₁/ω₂ is irrational for full coverage

3D MODE:
  g⃗(t) = R_z(ω₃t) · R_y(ω₂t) · R_x(ω₁t) · [0, 0, g₀]ᵀ
  Algorithm options: Constant Speed / Random Walk / Optimized RPM

── 4.2 QUATERNION ENGINE ───────────────────────────────────────────────────

MANDATORY: Use quaternions for ALL rotations. No Euler angles in
the core physics engine (Euler only for display/readout).

Quaternion representation: q = [w, x, y, z]
Quaternion multiplication: q₁ ⊗ q₂
Rotation application: v' = q ⊗ v ⊗ q*
Integration: q(t+dt) = q(t) + (dt/2)·Ω⊗q(t)
Where Ω = [0, ωx, ωy, ωz] — angular velocity quaternion

Normalize every step: q = q / |q|

── 4.3 RESIDUAL GRAVITY ────────────────────────────────────────────────────

g_res = √[(1/T·∫gx dt)² + (1/T·∫gy dt)² + (1/T·∫gz dt)²]

Quality threshold:
  - Good simulation: g_res < 0.01g
  - Excellent: g_res < 0.001g
  - ISS equivalent: g_res < 10⁻⁶g (flag in UI)

── 4.4 CENTRIFUGAL ACCELERATION AT SAMPLE ──────────────────────────────────

a_c = ω² · r
Where r = distance from rotation center to sample [meters]
      ω = angular velocity [rad/s]

Total disturbing acceleration:
a_total = √(g_res² + a_c²)

── 4.5 MOMENT OF INERTIA (FRAME) ───────────────────────────────────────────

For a thin circular ring (clinostat frame):
I = m · R²

For a solid cylinder (motor shaft):
I = (1/2) · m · R²

For combined system (parallel axis theorem):
I_total = I_cm + m · d²

── 4.6 STOKES SETTLING VELOCITY (BIOLOGICAL PAYLOAD) ───────────────────────

v_s = (2·r²·(ρ_p - ρ_f)·g_res) / (9·μ)

Where:
  r = cell/particle radius [m]
  ρ_p = particle density [kg/m³]
  ρ_f = fluid density [kg/m³]
  g_res = residual gravity [m/s²]
  μ = dynamic viscosity [Pa·s]

── 4.7 MOTOR POWER MODEL ───────────────────────────────────────────────────

P_mech = τ · ω
P_elec = V · I
η = P_mech / P_elec
P_heat = P_elec · (1 - η)

── 4.8 VIBRATION & RESONANCE ───────────────────────────────────────────────

Natural frequency of rotating frame:
f_n = (1/2π) · √(k/m)

Imbalance force:
F_imb = m_imb · e · ω²

FFT of accelerometer signal to detect resonance peaks.
Flag if any harmonic matches f_n ± 5%.

── 4.9 ROTATION MATRIX (DISPLAY LAYER) ─────────────────────────────────────

Rx(φ) = [[1,0,0],[0,cosφ,-sinφ],[0,sinφ,cosφ]]
Ry(θ) = [[cosθ,0,sinθ],[0,1,0],[-sinθ,0,cosθ]]
Rz(ψ) = [[cosψ,-sinψ,0],[sinψ,cosψ,0],[0,0,1]]

Combined: R = Rz·Ry·Rx (ZYX convention — document this choice)

COMMUNICATION PROTOCOL (Ethernet/WiFi over WebSocket):

── 5.1 PACKET STRUCTURE ────────────────────────────────────────────────────

Every packet sent/received must follow this structure:

{
  "packet_id": <uint32>,        // Incrementing packet counter
  "timestamp": <float64>,       // Unix timestamp ms
  "device_id": <string>,        // "AXIS_1" / "AXIS_2" / "AXIS_3"
  "command": <string>,          // "SET_RPM" / "STOP" / "STATUS" / "PING"
  "payload": {
    "rpm_setpoint": <float>,    // Desired RPM
    "direction": <int>,         // 1 = CW, -1 = CCW
    "rpm_actual": <float>,      // Encoder feedback
    "current_mA": <float>,      // Motor current draw
    "temp_C": <float>,          // Motor temperature
    "encoder_ticks": <uint32>,  // Raw encoder count
    "fault_flags": <uint8>      // Bitmask: bit0=stall, bit1=OC, bit2=enc
  },
  "crc32": <uint32>             // CRC32 of packet body
}

── 5.2 CONNECTION VALIDATION PROTOCOL ──────────────────────────────────────

STEP 1 — DISCOVERY:
  Software broadcasts UDP discovery packet on LAN subnet.
  Each motor controller responds with: device_id, firmware_version,
  capabilities, IP address, MAC address.

STEP 2 — HANDSHAKE:
  3-way handshake: SYN → SYN-ACK → ACK
  Exchange protocol version, max RPM, encoder resolution, current limits.
  Reject if version mismatch.

STEP 3 — CAPABILITY CHECK:
  Verify each subassembly supports required features.
  Log: max_rpm, encoder_ppr, current_limit, temperature_sensor, PID_support

STEP 4 — HEARTBEAT:
  Send PING every 500ms. Expect PONG within 200ms.
  3 missed heartbeats = CONNECTION LOST → alert + safe stop all motors.

STEP 5 — DATA STREAMING:
  Motor controllers push telemetry at 50Hz (configurable).
  Software timestamps and stores every packet in ring buffer.

── 5.4 CLINOSTAT MODE ALGORITHMS ───────────────────────────────────────────

1D MODE:
  ω₁ = user-defined (typically 1–10 RPM)
  Axis: horizontal (X-axis)
  Only Axis 1 motor active.

2D MODE (Classic RPM):
  ω₁ = user RPM (outer frame)
  ω₂ = auto-calculated = ω₁ · φ (golden ratio) for irrational ratio
  OR user manual input with irrationality warning if ratio rational.

3D MODE — Three sub-modes:
  a) CONSTANT SPEED: All 3 axes at fixed RPMs, irrational ratios
  b) RANDOM WALK: Axes change speed/direction randomly at intervals
     (interval = configurable, 5–60 seconds)
  c) OPTIMIZED RPM: Algorithm minimizes g_res at each step,
     dynamically adjusts ω₁, ω₂, ω₃ in real time

SECTION 11 — SIMULATION vs HARDWARE MODE (CRITICAL GAP #1)
═══════════════════════════════════════════════════════════════════════════════

The software MUST support TWO distinct operating modes, selectable from
the header bar with a clear toggle switch:

── 11.1 SIMULATION MODE ────────────────────────────────────────────────────

  Purpose: Full development and experiment design WITHOUT hardware

  Behavior:
  - Physics engine computes ALL values locally
  - Motor telemetry is SIMULATED using realistic models:
    • RPM actual = RPM setpoint ± random noise (σ = 0.2 RPM)
    • Current draw = f(RPM, load inertia) + noise
    • Temperature = ambient + thermal model (dT/dt = P_heat/C_thermal)
    • Encoder ticks = computed from RPM × PPR × dt
    • Packet latency = simulated with jitter (mean=5ms, σ=2ms)
  - Fault injection: UI button to simulate faults:
    • Motor stall (RPM drops to 0, current spikes)
    • Encoder failure (ticks freeze or jump)
    • Communication loss (packets stop)
    • Overtemperature (temp ramps to threshold)
  - Connection status shows: "SIMULATION MODE — No Hardware"
  - All CSV logging, plotting, and data export work identically
  - 3D visualization runs exactly the same as hardware mode

── 11.2 HARDWARE MODE ──────────────────────────────────────────────────────

  Purpose: Control real motor controllers over network

  Behavior:
  - Runs Section 5 connection validation protocol
  - Receives REAL telemetry from motor controllers
  - Physics engine uses ACTUAL encoder feedback for rotation state
  - Fault detection uses REAL sensor thresholds
  - E-STOP sends REAL stop command to hardware
  - Connection status shows: "HARDWARE — [IP] — [Latency]ms"

── 11.3 HYBRID MODE ────────────────────────────────────────────────────────

  Purpose: Test software with partial hardware

  Behavior:
  - Connected axes use real hardware telemetry
  - Disconnected axes fall back to simulation
  - UI clearly marks which axes are REAL vs SIMULATED
  - Color coding: Real axis = green border, Simulated = amber border

  │   ├── faultInjector.js       ← Manual fault injection for testing
  │   └── modeManager.js         ← SIM/HARDWARE/HYBRID mode controller

SECTION 14 — PS4 FLIGHT PROFILE COMPARISON (CRITICAL GAP #4)
═══════════════════════════════════════════════════════════════════════════════

The entire purpose of ground clinostat simulation is to predict and
validate against REAL microgravity aboard PS4. This section defines
how the software compares clinostat data with actual flight data.

── 14.1 FLIGHT DATA IMPORT ─────────────────────────────────────────────────

  Supported formats:
  - CSV (standard accelerometer data)
  - JSON (structured mission data)
  - TXT (tab/space delimited)
  - HDF5 (if Python backend available)

  Required columns (auto-detect or manual mapping):
  - timestamp (Unix ms or ISO 8601)
  - ax, ay, az — accelerometer readings in m/s² or g
  - (optional) temperature, mission_phase, altitude

── 14.2 COMPARISON ANALYSIS ────────────────────────────────────────────────

  Metrics computed when flight data is loaded:

  1. GRAVITY MAGNITUDE COMPARISON:
     |g_clinostat(t)| vs |g_flight(t)|
     Plot both as overlay on same time axis

  2. TIME-AVERAGED RESIDUAL COMPARISON:
     g_res_clinostat vs g_res_flight over sliding windows

  3. PEARSON CORRELATION COEFFICIENT:
     r = corr(g_clinostat_series, g_flight_series)
     Display: r value + p-value + confidence interval

  4. POWER SPECTRAL DENSITY (PSD):
     PSD of clinostat vs PSD of flight accelerometer data
     Overlay frequency-domain plots to compare vibration signatures

  5. GRAVITY VECTOR COVERAGE:
     - Plot g-vector tips on unit sphere for both datasets
     - Compute spherical coverage percentage
     - Ideal clinostat: uniform coverage = 100%
     - Compare coverage uniformity between clinostat and flight

  6. FIDELITY SCORE:
     Composite metric (0-100%):
     F = w1·(1 - |g_res_clino - g_res_flight|/g₀) +
         w2·r_correlation +
         w3·coverage_uniformity +
         w4·(1 - PSD_difference)
     Weights: w1=0.3, w2=0.3, w3=0.2, w4=0.2 (configurable)

── 14.3 MISSION TIMELINE SYNC ──────────────────────────────────────────────

  - Import PS4 mission timeline with phases:
    Launch / Ascent / Orbit Insertion / Microgravity Ops / Re-entry
  - Map clinostat experiment phases to mission phases
  - Visual Gantt chart overlay: clinostat vs flight timeline
  - Time offset correction for synchronization

── 14.4 COMPARISON UI ──────────────────────────────────────────────────────

  - Split-screen mode: Clinostat data (left) vs Flight data (right)
  - Overlay mode: Both datasets on same plot
  - Fidelity Score dashboard: large number + color (green/yellow/red)
  - Report generator: export comparison as PDF/HTML summary

SECTION 15 — MULTI-SAMPLE POSITION ANALYSIS (CRITICAL GAP #5)
═══════════════════════════════════════════════════════════════════════════════

Real payloads contain MULTIPLE samples at DIFFERENT positions within the
payload chamber. Each position experiences DIFFERENT residual gravity
due to centrifugal effects. This section ensures the software analyzes
the ENTIRE payload volume, not just a single point.

── 15.1 ANALYSIS GRID ──────────────────────────────────────────────────────

  Generate a 3D grid of analysis points within the payload chamber:
  - Grid resolution: configurable (default: 5mm spacing)
  - Grid shape: matches payload geometry (sphere, cylinder, box)
  - Each grid point tracks:
    • Position vector r⃗ from rotation center [m]
    • Instantaneous g⃗(t) at that point
    • Time-averaged g_res at that point
    • Centrifugal acceleration: a_c = ω² × |r⃗|
    • Total disturbing acceleration at that point
    • Stokes settling velocity for sample at that point

── 15.2 VOLUME HEATMAP ─────────────────────────────────────────────────────

  On the 3D model:
  - Render payload chamber as transparent shell
  - Inside: color-coded 3D point cloud or volumetric map
  - Color scale: g_res magnitude
    Green (< 0.001g) → Yellow (< 0.01g) → Red (> 0.01g)
  - User can slice through volume: X-plane, Y-plane, Z-plane
  - Cross-section view: 2D heatmap of selected slice
  - Interactive: click any point to see exact values

── 15.3 OPTIMAL SAMPLE PLACEMENT ───────────────────────────────────────────

  Algorithm to find the BEST positions for samples:
  - Input: number of samples, sample radius, min inter-sample distance
  - Output: optimal positions that minimize max(g_res) across all samples
  - Constraint: all positions must be inside payload chamber
  - Method: gradient descent or grid search over candidate positions
  - Display: highlighted spheres at optimal positions in 3D view

── 15.4 STATISTICS ──────────────────────────────────────────────────────────

  For the entire payload volume:
  - Mean g_res across all grid points
  - Max g_res (worst case) and its position
  - Min g_res (best case) and its position
  - Volume percentage where g_res < threshold (user-defined)
  - Uniformity index: std(g_res) / mean(g_res)
  - Histogram: distribution of g_res values across volume

── 15.5 COORDINATE SYSTEM DEFINITION ───────────────────────────────────────

  MANDATORY CONVENTION (applies to ALL sections):
  - RIGHT-HAND coordinate system
  - Z-axis = UP (opposite to gravity in lab frame)
  - X-axis = horizontal, pointing toward operator
  - Y-axis = horizontal, pointing left (right-hand rule)
  - Gravity vector in lab frame: g⃗_lab = [0, 0, -g₀]
  - This is the ENU (East-North-Up) convention with Z=Up

  Frame rotation axes:
  - Frame 1 (inner): rotates about X-axis
  - Frame 2 (middle): rotates about Y-axis
  - Frame 3 (outer): rotates about Z-axis
